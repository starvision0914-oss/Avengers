대량(수천~수만 건) 상품명 최적화 작업을 API 키 없이 대화 세션 안에서 직접 처리할 때 쓴 방법. 사용자가 "API 키 대신 여기서 네가 진행해줘"라고 명시적으로 요청해 이 방식을 채택했고 잘 작동함.

## 핵심 원칙: 창작이 아니라 "스캔 후 예외처리"
- 전체를 하나하나 창작(rewrite)하려 하면 수만 건 규모에서 불가능. 대신:
  1. 금칙어/위험 키워드로 **먼저 스캔**해서 진짜 손봐야 할 항목만 추려낸다 (실측: 3,464건 중 26건, 15,395건 중 151+38건 정도로 실제 위험군은 전체의 1~5% 수준).
  2. 나머지 대다수(90%+)는 이미 클린한 상태 → 원본 유지가 정답인 경우가 많음. "다 바꿔야 한다"는 가정을 버릴 것.
  3. 애매한 브랜드/캐릭터 건은 hulk API의 detailContent(보통 이미지 1장)를 다운로드해 Read 툴로 직접 봐서 라이선스 판정(제조사·상품코드 명시 여부로 판단).
  4. 50byte 초과처럼 기계적으로 처리 가능한 문제(중복단어 제거+뒷부분 트림)는 스크립트로 일괄 처리.
  5. 카테고리 기반 키워드 추가처럼 "정답이 있는" 보강은 룰 기반으로 스케일링 가능하지만, **셀러의 카테고리 오분류가 섞여있을 수 있어** 카테고리-상품명 어휘 연관성 체크(같은 리프카테고리 다른 상품들과 토큰 겹치는지) 필터를 반드시 거칠 것 — 안 거르면 엉뚱한 키워드가 붙는 사고 발생(칫솔에 "작업 수공구" 붙는 식).

## 적용/실행 시 주의
- 실제 라이브 반영(hulk API PUT 등)은 되돌리기 어려운 작업 — 반드시 소량(5건) 테스트 후 전체 실행.
- 11번가는 `guard.preflight(wait=True)`로 락을 걸어야 함. verify/광고전략 등 다른 예약작업과 순번대기 발생할 수 있고 몇 시간 걸릴 수 있음 — 진행 중인 배치를 죽이지 말고 자동 대기하도록 설계.
- 속도(--sleep) 조정은 사용자에게 트레이드오프(계정 이상탐지 위험 vs 시간)를 명시하고 선택받을 것 — 확인 없이 임의로 빠르게 돌리지 말 것.

**Why:** 15,395건(jinag7460) 작업에서 이 방식으로 API 비용 없이 몇 시간 내 처리 완료. [[project_11st_jinag7460_name_opt]]
**How to apply:** 다른 계정/플랫폼(지마켓 등)에도 동일 패턴 적용 가능 — "전수 스캔→위험군만 정밀처리→기계적 처리는 스크립트→카테고리 키워드는 연관성 필터" 순서를 기본 틀로 삼을 것.
